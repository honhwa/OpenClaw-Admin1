## 质量验收标准

### P0 需求验收标准

| 需求 ID | 验收标准 |
|--------|---------|
| R-01 | 1. 支持创建/编辑/删除用户<br>2. 三种角色（管理员/操作员/只读）权限隔离<br>3. 敏感操作需二次确认<br>4. 所有操作有审计日志 |
| R-02 | 1. 通知中心可查看所有历史通知<br>2. 支持配置 3 种告警渠道（飞书/钉钉/邮件）<br>3. 5 种核心告警场景可触发（Gateway 断开/任务失败/Agent 崩溃/Token 超阈值/计划外支出） |
| R-03 | 1. 可创建多 Agent 场景<br>2. 可委派任务给指定 Agent<br>3. Agent 间可传递消息<br>4. 任务执行结果可查看 |
| R-04 | 1. 可添加角色到虚拟场景<br>2. 角色可在场景中移动<br>3. 角色可进入不同区域互动<br>4. 角色间可实时通信 |

## 附录：反共识建议

1. **Office/MyWorld 不要追求完美** —— 先做最小可用链路，验证用户真实需求后再完善
2. **告警不要贪多** —— 第一期只做 5 种核心告警，不做"告警规则引擎"
3. **RBAC 不要过度设计** —— 3 个角色（管理员/操作员/只读）足够初期使用
4. **Webhook 优先于具体告警渠道** —— Webhook 最通用，用户可自己转发到任意平台
